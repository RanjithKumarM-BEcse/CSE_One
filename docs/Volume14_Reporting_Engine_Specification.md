# CSE One - Volume 14
## Reporting Engine & Document Generation Platform

### 1. Reporting Engine Overview
The Reporting Engine is a centralized, robust subsystem within CSE One dedicated exclusively to the generation, formatting, and delivery of official institutional documents. It decouples document generation from the dashboard analytics (Volume 13), ensuring that heavy PDF rendering or massive Excel exports do not degrade the performance of the primary user interfaces. It acts as the singular source of truth for all printable and exportable data, enforcing strict branding and security guidelines across all roles.

### 2. Reporting Architecture
The architecture is designed as an asynchronous, queue-based pipeline.

```mermaid
graph TD
    subgraph Client Layer
        Web[PWA Frontend]
    end

    subgraph API Gateway
        API[Report API]
    end

    subgraph Core Backend
        Access[Role Validator]
        Builder[Report Builder]
    end

    subgraph Generation Cluster
        Queue[(Task Queue - Redis/Celery)]
        PDF[PDF Engine (Headless Chrome/WeasyPrint)]
        Excel[Excel Engine (openpyxl)]
    end

    subgraph Data Sources
        DB[(PostgreSQL)]
        Analytics[(Materialized Views)]
    end

    subgraph Storage & Delivery
        S3[(Object Storage / Temp Cache)]
    end

    Web -->|POST Request| API
    API --> Access
    Access --> Builder
    Builder -->|Enqueue Job| Queue
    Queue --> PDF
    Queue --> Excel
    PDF -->|Fetch Data| DB
    PDF -->|Fetch Data| Analytics
    PDF -->|Store| S3
    Excel -->|Store| S3
    S3 -->|Signed URL| Web
```

### 4. Report Builder
A dynamic interface primarily utilized by Administrators to construct custom views without requiring developer intervention.
- **Select Data Domain:** (e.g., Attendance, Leaves).
- **Column Picker:** Drag and drop required columns (RegNo, Name, % Present).
- **Filters:** Apply dynamic SQL `WHERE` clauses via UI (e.g., `attendance_percent < 75`).
- **Save Configuration:** Admins can save this configuration as a "Custom Template" for future one-click generation.

### 5. Report Preview
Before committing backend resources to a heavy PDF/Excel generation, the frontend leverages a lightweight HTML/CSS preview.
- **Data Pruning:** The preview API endpoint fetches only the first 25 rows (`LIMIT 25`).
- **Visual Validation:** The user sees exactly how the headers, columns, and data will align on an A4 layout.
- **Actions:** The preview modal offers [ Download PDF ], [ Download Excel ], or [ Cancel ].

### 6. Report Scheduling
Administrators can automate recurring reports.
- **Cron Engine:** Built on standard CRON syntax.
- **Use Case:** "Generate the *Defaulters List (<75%)* every Friday at 17:00 and deliver it."
- **Execution:** The `SchedulingService` wakes up, triggers the generation pipeline as if the Admin clicked the button, and routes the output to the Delivery Service.

### 7. Report Delivery
- **Direct Download:** Synchronous file stream for small reports (< 100 rows).
- **Notification Link:** For large reports, the UI returns a 202 Accepted. The user receives an in-app notification when the file is ready, containing a secure, signed URL (S3 or local temp storage).
- **Future-Ready:** Email integration payload structures are defined, awaiting SMTP configuration.

### 8. Role-Based Reporting
The engine enforces data isolation at the query level before a single pixel is rendered.
- **Student:** Hardcoded filter `student_id = {jwt.sub}`.
- **Professor:** Hardcoded filter `period_id IN (SELECT id FROM periods WHERE professor_id = {jwt.sub})`.
- **Faculty Advisor:** Hardcoded filter `student_id IN (SELECT id FROM student_advisors WHERE fa_id = {jwt.sub})`.
- **Admin:** Bypass filters; global access.

### 9. API Specifications
- `POST /api/v1/reports/preview`: Returns HTML/JSON of the first 25 rows based on filters.
- `POST /api/v1/reports/generate`: Submits a generation job. Returns `{ "job_id": "123", "status": "processing" }`.
- `GET /api/v1/reports/status/{job_id}`: Polls for job completion. Returns signed URL upon success.
- `GET /api/v1/reports/history`: Returns previously generated reports for the current user.
- `POST /api/v1/reports/schedules`: Creates a new CRON-based recurring report.

### 10. Backend Services
- **ReportGenerationService (Facade):** Handles API requests, role validation, and job queuing.
- **TemplateService:** Injects dynamic data into the HTML/Jinja2 templates required for PDF rendering.
- **ExportService:** Handles the actual binary creation (using libraries like `WeasyPrint` for PDF and `openpyxl` for Excel).
- **DeliveryService:** Manages the generation of short-lived, signed URLs for secure downloading.
- **SchedulingService:** Manages recurring CRON triggers.

### 11. Frontend Specifications
- **Report Dashboard:** A clean, grid-based interface listing available templates based on the user's role.
- **Wizard Flow:** Generating a report is a step-by-step wizard: 1. Select Template -> 2. Apply Filters -> 3. Preview -> 4. Export.
- **Loading States:** Uses an indeterminate progress bar while polling `/status/{job_id}` to assure the user the report is building in the background.

### 12. Business Rules
- **Immutability:** Reports reflect the data at the exact millisecond of generation. They do not auto-update. If data changes, a new report must be generated.
- **Mandatory Branding:** It is impossible to bypass the S.A. Engineering College headers and footers for any PDF output.
- **Watermarking:** Draft or preview reports contain a diagonal "PREVIEW" watermark to prevent them from being used officially.

### 13. Security Strategy
- **Signed URLs:** Download links expire after 15 minutes to prevent unauthorized sharing of sensitive data.
- **Row-Level Security:** The Role Validator injects the user's ID directly into the base SQL query view, ensuring a Professor can never generate a PDF containing another Professor's class data, even if they manipulate API payload parameters.

### 14. Audit Logging
The `audit_log` captures every interaction with the Reporting Engine.
- **Log Payload:** `ACTION: REPORT_GENERATED`, `ACTOR: {User_ID}`, `METADATA: { "template": "Section_Attendance", "filters": {"section": "A", "year": "III"} }`.
- **Why:** To ensure accountability if a printed report is misused or leaked.

### 15. Performance Strategy
- **Asynchronous Execution:** No HTTP request waits for a PDF to render if it takes longer than 3 seconds. The architecture relies entirely on the Task Queue (Celery).
- **Streaming Output:** When downloading massive CSVs (e.g., all 4 years of attendance data), the backend streams the database cursor directly to the HTTP response to prevent memory exhaustion (OOM crashes).

### 16. Testing Strategy
- **Export Validation:** Automated tests parse the generated `.xlsx` binary to assert that Cell A1 contains "S.A. ENGINEERING COLLEGE".
- **Permission Tests:** Asserts that an FA attempting to generate a report for a Section they do not advise receives a `403 Forbidden` or an empty report.
- **Load Testing:** Spawns 100 concurrent Celery workers generating PDFs simultaneously to ensure the server CPU does not bottleneck the core transaction system.

### 17. Reporting Engine Architecture Decision Record (ADR)
- **ADR-RE-001: HTML-to-PDF over Native PDF Drawing:** Chosen (using tools like WeasyPrint or Puppeteer) because it allows developers to build report templates using standard CSS/HTML, drastically reducing the time required to design complex institutional tables compared to native drawing libraries (like ReportLab).
- **ADR-RE-002: Asynchronous Generation for All Reports:** Chosen to create a uniform frontend architecture. Even if a report is small and could be generated synchronously, treating all reports as async jobs eliminates timeout edge cases and simplifies UI state management.
- **ADR-RE-003: Centralized Report Definitions:** Chosen over letting individual frontend dashboards build their own exports. This guarantees that an "Attendance Report" generated by a Student, Professor, or Admin utilizes the exact same mathematical queries and visual branding.
